## 3. 学习过程

### 3.1 HTTP 基础回顾
- HTTP（HyperText Transfer Protocol）是应用层协议，用于客户端和服务端通信
- 基于请求-响应模型，无状态

### 3.2 HTTP 请求结构
```
请求行 + 请求头 + 空行 + 请求体
```
- 请求行包含：方法、URL、HTTP 版本
- 请求头是键值对，提供请求元信息

### 3.3 请求方法
- GET：获取资源（参数在 URL）
- POST：提交数据（参数在请求体）
- PUT：完整更新资源
- PATCH：部分更新资源
- DELETE：删除资源

### 3.4 HTTP 响应结构
```
状态行 + 响应头 + 空行 + 响应体
```
状态码分类：
- 1xx：信息性
- 2xx：成功（200 OK, 201 Created）
- 3xx：重定向（301, 302, 304）
- 4xx：客户端错误（400, 401, 403, 404）
- 5xx：服务端错误（500, 502, 503）

### 3.5 HTTPS 原理
- HTTPS = HTTP + TLS/SSL
- TLS 握手过程：
  1. ClientHello / ServerHello 协商版本和算法
  2. 服务端发送证书（包含公钥）
  3. 客户端验证证书
  4. 客户端发送用公钥加密的 Pre-Master Secret
  5. 双方生成会话密钥，开始加密通信

### 3.6 HTTP/2 和 HTTP/3
- HTTP/2：多路复用、Header 压缩、服务器推送、二进制分帧
- HTTP/3：基于 QUIC，0-RTT，解决队头阻塞

## 4. 学习内容

### 4.1 HTTP 请求行
```
GET /api/users HTTP/1.1
```

### 4.2 HTTP 响应状态码速记
- 200 系列：搞定 ✅
- 300 系列：换个地方 🔄
- 400 系列：你有问题 ❌
- 500 系列：我有问题 💥

### 4.3 HTTPS 加密方式
- 非对称加密：公钥/私钥，用于密钥交换
- 对称加密：会话密钥，用于实际数据加密
- 数字证书：由 CA 机构签发，证明身份

## 6. 心得感悟

今天学习 HTTP/HTTPS 协议，有一种"原来如此"的感觉。之前开发中经常看到这些概念，但从来没有系统地梳理过。

**收获**：
1. HTTP 是无状态的，每次请求都是独立的
2. HTTPS 通过 TLS 握手实现了加密和身份认证
3. HTTP/2 的多路复用很好地解决了队头阻塞问题
4. 状态码不是孤立的，是有逻辑分类的

**思考**：
为什么 HTTPS 握手需要非对称加密，而不是直接用对称加密？
- 因为对称加密需要双方共享密钥，但网络上传输密钥不安全
- 非对称加密可以安全地传输对称密钥

这个问题在之前的项目中没有深入思考过，现在终于理解了底层原理。
